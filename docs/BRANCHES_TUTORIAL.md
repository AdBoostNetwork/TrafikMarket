# BRANCHES_TUTORIAL.md

## Что такое ветка в Git (простыми словами)

Ветка (branch) — это отдельная "линия разработки".

- `main` — обычно стабильная основная ветка.
- В отдельной ветке вы делаете свою задачу, не ломая `main`.
- Когда задача готова, изменения вливаются обратно в `main` через Pull Request.

Идея простая: **одна задача = одна ветка**.

---

## Зачем это нужно в нашем проекте

У нас несколько участников. Если все пушат прямо в `main`, легко получить:

- конфликты,
- случайные поломки,
- потерю контекста "кто что менял".

Поэтому рабочий процесс такой:

1. Обновиться от `main`.
2. Создать свою ветку.
3. Сделать изменения.
4. Закоммитить.
5. Запушить ветку в `origin`.
6. Открыть PR в `main`.

---

## Базовый сценарий (копируйте как есть)

> Выполнять из корня проекта `TrafikMarket`.

```bash
# 1) Перейти в проект
cd TrafikMarket

# 2) Проверить, что нет незакоммиченных изменений
git status

# 3) Переключиться на main и подтянуть свежие изменения
git checkout main
git pull origin main

# 4) Создать новую ветку под задачу и сразу на неё переключиться
git checkout -b feature/short-task-name
# пример: git checkout -b fix/create-announ-await

# 5) Сделать изменения в коде, затем проверить
git status

# 6) Добавить файлы в commit
git add .
# или точечно: git add backend/app_backend/endpoints.py

# 7) Создать commit
git commit -m "fix: add await for create_announ"

# 8) Первый push ветки на GitHub
git push -u origin feature/short-task-name
```

После этого на GitHub появится ваша ветка. Создайте Pull Request: **base = `main`**, **compare = ваша ветка**.

---

## Как назвать ветку

Рекомендуемый формат:

- `feature/...` — новая функциональность
- `fix/...` — исправление бага
- `refactor/...` — рефакторинг без изменения логики
- `docs/...` — документация

Примеры:

- `feature/orders-page-api`
- `fix/profile-null-check`
- `docs/branches-tutorial`

Используйте короткие и понятные имена.

---

## Если работа уже в ветке и нужно допушить

```bash
# проверить текущую ветку
git branch --show-current

# добавить новые изменения
git add .
git commit -m "feat: continue orders endpoint"

# пуш в ту же ветку
git push
```

---

## Если случайно начали работать в main

```bash
# сохранить текущие изменения в новую ветку
git checkout -b fix/my-task-from-main

# запушить её
git push -u origin fix/my-task-from-main
```

То есть изменения не потеряются: вы просто переносите их в новую ветку.

---

## Частые ошибки

### 1) Пытаюсь `git push`, а пишет "no upstream branch"

Сделайте первый push так:

```bash
git push -u origin <имя-ветки>
```

### 2) Перед `git pull` открывается редактор и вы не понимаете, как выйти

Если это `vim`:

- `Esc`
- сохранить и выйти: `:wq` + Enter
- выйти без сохранения: `:q!` + Enter

### 3) Конфликт при merge/rebase

Это нормально при командной работе. Не пушьте "как есть". Разрешите конфликт, проверьте проект, затем commit/push.

---

## Мини-чеклист перед PR

- Ветка не `main`
- Изменения только по вашей задаче
- Код запускается/тесты проходят (если есть)
- Понятное сообщение commit
- PR в `main`

---

## Коротко в одну строку

**Всегда работаем так: `main` обновили -> новая ветка -> изменения -> commit -> push ветки -> PR в `main`.**
